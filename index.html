Análise Exploratória/Preliminar/Inicial

Verificar o maior número possível de informações que podem ser extraídas de uma análise exploratória inicial das Temperaturas das salas da Elipse-RS no mês de junho de 2017.

  • ADM_Temperature
  • EPMDev_Temperature
  • PowerDev_Temperature
  • E3Dev_Temperature

A ferramenta ideal para realizar este exercício é o Chart Analysis do EPM Studio, devido as facilidades e flexibilidade de seleção de variáveis e períodos para visualização em um gráfico de tendência.

Neste ambiente também é possível criar Bookmarks de períodos de interesse, exportar os dados para arquivos no formato CSV ou mesmo criar um Dataset para detalhamento das análises no ambiente Dataset Analysis, tudo dentro de uma mesma ferramenta, o EPM Studio.

Análise preliminar – “primeira impressão” do sistema. Esta “primeira impressão” é que orientará um posterior aprofundamento das análises, além de eventualmente levantar questões que nem haviam sido cogitadas no início do processo de análise.

Palavra chave: DISCERNIMENTO

Numa "primeira olhada" percebe-se a periodicidade dos dados das medidas de temperatura. Fazendo um Zoom In, verifica-se que são ciclos diários.

Além disso, também é possível identificar ruídos de medida nestes dados, indicando que ainda não devem ter passado por pré-processamentos e/ou agregações, provavelmente provenientes diretamente de uma base de dados que armazenou estes valores a partir de um instrumento de medição de temperatura.

Nota: Este gráfico de tendências apresenta a sequência de valores que foram armazenados na medida em que foram sendo recebidos pelo sistema de coleta de dados de processo. Cada valor corresponde a: um valor (medido pelo instrumento de medida), um timestamp (algumas vezes traduzido por "estampa/selo de tempo") e uma qualidade (relacionada a qualidade da medida realizada pelo instrumento).

Ainda é possível identificar uma amplitude típica nos dados, que dão uma ideia da amplitude térmica do período - traduzindo para a dimensão do processo em análise ☺.

Nota-se também que alguns valores são muito discrepantes dos demais, partindo-se da premissa que a temperatura de uma sala não sofre variações tão bruscas e nem de tal magnitude como se constata nestes valores - "cara típica de ser um" outlier.

Vale salientar que a remoção/tratamento de outliers não é uma tarefa trivial, ou seja, não existe uma "fórmula mágica" a ser aplicada em todos os casos. Aqui entra muito do conhecimento do processo em questão por parte do analista para a decisão de como proceder para o pleno atendimento das suas necessidades de análise, bem como ter ciência das implicações das suas decisões tomadas nesta etapa.

Apenas a título de exemplo, uma forma de identificação de outliers muito utilizada baseia-se no intervalo/faixa interquartis (IIQ), onde todos os valores que estiver fora de uma faixa pré-definida, são identificados como outliers. Tipicamente utiliza-se o valor de 1,5 * IIQ acima do terceiro quartil ou abaixo do primeiro quartil como faixa de valores não conformes/expúrios, ou seja, valores nestas regiões são considerados outliers.

Por fim, é possível se ter alguma ideia sobre as temperaturas mínimas, máximas e médias, bem como identificar um comportamento diferenciado no perfil das temperaturas nos finais de semana - o que é esperado por aqueles que conhecem os ciclos de operação deste local, que ocorrem apenas nos dias úteis e em horário comercial.

Ciclo PDCA - Melhoria Contínua

Para implementar o PDCA é fundamental ter DADOS para:


  • planejar e executar as ações
  • verificar os resultados (a partir de referências)
  • implementar os ajustes
  • monitorar indicadores

Existe uma relação de dependência direta entre o Ciclo de Melhoria Contínua(PDCA) e os Dados de Processo.

Tentar implementar um ciclo de melhoria continuada sem Dados de Processo é como tentar pilotar um avião sem a devida instrumentação, por isso é necessário, antes de tudo, promover uma transformação digital na empresa para poder se iniciar qualquer trabalho nesta direção.

Ciclo PDCA - "malha de controle"

Visão esquemática do ciclo PDCA na forma de malha de controle feedback.

"Iniciando" por um planejamento para levantar os dados necessários e posteriormente um planejamento das ações a serem tomadas após as devidas análises. Aplicam-se as ações necessários sobre o negócio (processo) conforme o planejamento prévio e segue-se para o monitoramento dos resultados.

A etapa de monitoramento deve, sempre que possível, ser automatizada, pois na prática os processos sofrem variações naturais, seja de matérias primas, seja por desgastes de equipamentos, etc. - obrigando que todos seus ativos/recursos estejam sempre sendo acompanhados via indicadores predefinidos a fim de proporcionar uma operação adequada/apropriada, dentro das especificações estabelecidas nas etapas anteriores de planejamento.

Análises - Visão Geral

Esta é uma visão esquemática sobre o processo de análise de dados de processo.

Notar que o que "chamamos simplesmente de análises", na verdade é um processo relativamente complexo, envolvendo diversas etapas como, por exemplo: a seleção dos dados em uma análise preliminar, o processamento/tratamento dos dados previamente selecionados, a "análise" em si utilizando ferramentas de visualização para promover insights e, por fim, a apresentação dos resultados obtidos e eventualmente a automatização, na medida do possível, deste processo ou partes dele.

A automatização de partes deste processo de análise, ou, na medida do possível, do processo como um todo, é o que se busca cada vez mais para a implementação de um Ciclo de Melhoria Cóntínua, uma vez que o volume de dados e informações para serem processados vem se tornando cada vez maior, chegando cada vez mais rápido, além da demanda crescente por informações consolidadas e em tempo real para dar suporte às decisões. Este cenário é que define os requisitos de infraestrutura fundamentais para fazer frente a um mercado altamente competitivo → este é justamente o mote do EPM Processor.

Essencialmente, os dados de processo são do tipo Séries Temporais. As informações típicas que podem ser extraídas a partir deles (dependendo da sua característica) são de natureza relacionada ao processo em si, além de informações obtidas a partir da aplicação da estatística descritiva sobre mesmos.

Séries temporais

Tipos de informações

  • Valor Value
  • Estampa de tempo Timestamp
  • Qualidade Quality

Natureza da informação

  • Discreta digital
  • Contínua analógica

As Séries Temporais de Dados de Processo tem sempre associado a cada leitura: um valor, uma estampa de tempo (timestamp) e uma qualidade que, em geral, é produzida pelo instrumento de medida e está relacionada com a qualidade da informação da medição naquele ponto.

A natureza das variáveis podem ser contínuas (temperaturas, pressões, etc.) ou discretas (setpoints, on-off, etc.) - geralmente designadas, respectivamente, por "analógicas" e "digitais" no jargão de Controle e Automação de Processos.

Ao ser definida a natureza da variável, deve-se também ter especial atenção à amostragem da mesma, pois mesmo uma variável de natureza contínua, se for coletada a "elevados" intervalos de tempo, na prática tenderão a ter características de uma variável discreta, uma vez que, até chegar uma nova medida, a melhor informação que temos é o valor medido do tempo anterior - ou isso, ou um uma implementação com Filtro de Kalman para estimativas mais interessantes !!! .☺

Para mais informações sobre Filtro de Kalman e sua utilização na prática, recomendamos a tese de Doutorado de Leonardo A. Scardua - USP 2015:Sintonia Automática do Filtro de Kalman Unscented.

Para maiores informações sobre a amostragem, recomendamos pesquisar sobre teorema da amostragem de Nyquist–Shannon, ou simplesmente Teorema de Nyquist.

Para um aprofundamento maior na teoria sobre sinais e prática no uso sobre dados de processo, recomendo o livro Introdução à Identificação de Sistemas. Técnicas Lineares e não Lineares Aplicadas a Sistemas. do prof. Luis Antonio Aguirre:"Introdução à Identificação de Sistemas"

Análise Exploratória (Séries temporais)

Identificação da dinâmica do processo

Em grande parte das análises sobre dados de processo, o que se procura é identificar como as variáveis de interesse para controle respondem às variáveis utilizadas para controlá-las. Geralmente estas informações são obtidas a partir de experimentos denominados de Step Tests e todas suas variações possíveis da aplicação deste método.

Estes experimentos são muito comuns para a Identificação da Dinâmica do Processo (Identificação de Sistemas Dinâmicos), produzindo alguma forma de modelo que relacione estas variáveis, para então utilizar no comissionamento de controladores PID (e suas possíveis combinações) ou outras aplicações.

Assista ao vídeo: PID Control in Python - APMonitor.com. Nele é apresentado de maneira super didática e prática, como proceder com este método e posterior identificação da dinâmica e comissionamento de um -controlador PID tudo utilziando a linguagem Python!!!

Seleção de período & tratamento prévio dos dados

Uma vez identificadas as variáveis de interesse e os períodos que "parecem conter" os dados com as "informações desejadas", parte-se para uma etapa de pré-processamento destes dados, removendo outliers, tratando dados que faltam (Missing Data), removendo ruídos de medidas aplicando técnicas de filtragem dos dados, etc. para então seguir a etapa de um maior detalhamento (=quantificação), no caso do exemplo, a identificação da dinâmica para comissionar um controlador PID.

Sistemas de Gerenciamento de Dados de Processo (PIMS) geralmente apresentam um conjunto de propriedades relacionadas às variáveis de processo que ele gerencia (coleta, organiza/padroniza, armazena e disponibiliza consultas).

É de grande relevância que o analista possa dispor destas informações durante a etapa de planejamento dos experimentos ou, pelo menos, durante o processo de análise para saber se os dados em questão possuem características que possam, eventualmente, até invalidar a análise que se pretendia realizar.

No caso do exemplo Step Test comentado anteriormente, é fundamental que o sistema de coleta destes dados consiga capturar informações dinâmicas dos mesmos, o que pode implicar na necessidade de desabilitar a aplicação de algum algoritmo de compressão de dados com descarte durante os testes, para evitar que informações preciosas possam ser perdidas em um teste - que muitas vezes são onerosos, seja em termos de custo e/ou execução.

Esta prática de desabilitar os mecanismos de compressão dos dados durante testes em plante é bastante comum para quem trabalha com a Identificação de Sistemas para aplicações/soluções de Controle Avançado e Otimização em Tempo Real.

Para se ter uma ideia um pouco mais clara de todo o processo que envolve a proposição de uma solução envolvendo Controle Avançado, recomendamos ler a dissertação de mestrado de Lisandra Silveira da Silveira - UFRGS/2012, entitulada: Desenvolvimento de uma estratégia de controle avançado para uma Unidade Recuperadora de Propano.

ATENÇÃO na Coleta de Dados

Em geral, os softwares PIMS recebem dados por exceção.

Este mecanismos tem impactos diretos sobre a redução do tráfego na rede & nas Consultas aos Dados, que é o que mais interessa para "fazer" engenharia!!!

Em geral, os PIMS coletam os dados por exceção, ou seja, apenas os novos valores medidos do processo que estiverem fora de uma banda pré-configurada (que pode ser = 0) tendo como referência o último valor coletado, são encaminhados para serem processados e, eventualmente armazenados definitivamente na base de dados.

Por conta disso, estes dados costumam não estar igualmente espaçados, sendo necessário aplicar uma interpolação nos mesmos para a aplicação da maioria dos algoritmos comumente utilizados em análises.

O padrão OPC UA (amplamente utilizado na comunicação entre sistemas modernos) já tem esta consulta interpolada definida, levando em consideração inclusive as qualidades associadas às medidas, podendo ser utilizada diretamente durante o processo de análise.

Vale ressaltar que, além desta consulta aplicando uma agregação do tipo Interpolative, outras formas de agregar (= processamento dos dados na consulta), também estão definidos no padrão OPC UA (OPC Foundation).

ATENÇÃO na Compressão dos Dados

  • Compressão sem descarte de dados
  • Compressão com descarte de dados existe um compromisso em redução do espaço de armazenamento e reconstituição aceitável do sinal original

Os Sistemas de Gerenciamento de Dados de Processo surgiram na década de 80 a partir de uma demanda por informações do Processo Produtivo que, por sua vez, produz um elevado volume de dados a relativamente altas taxas - algumas das características do que designamos hoje de Big Data.

No passado, porém, com as tecnologias que se tinha disponível e seus custos associados, foram implementadas medidas alternativas para "dar conta" destas demandas, dentre elas a incorporação de algoritmos que, em tempo real, "decidiam" quais valores dos dados precisavam ser armazenados para posteriormente reconstituir o sinal original conforme uma dada necessidade, enquanto que os demais valores eram descartados, economizando assim os recursos disponíveis.

Hoje, apesar dos custos de armazenamento serem menores, não justificando muitas vezes o uso destes algoritmos, não devemos esquecer que o volume de informações chegando também aumentou, motivo pelo qual estes algoritmos ainda continuam sendo necessários e aplicados em muitos casos.

Para um analista de dados de Processos Produtivos (em geral um Engenheiro) é fundamental ter ciência desta questão, pois a aplicação destes algoritmos, quando bem comissionados (=ajustados/definidos), tende a preservar as informações de tendências centrais e dispersões que, em geral, são informações de interesse gerencial, porém costumam descartar informações dinâmicas que, na maioria das vezes, são justamente as informações de interesse de tático - de processo - o que o engenheiro procura saber!

ATENÇÃO na Compressão dos Dados

Dependendo das configurações utilizadas no Algoritmo de Compressão, as informações dinâmicas podem ser perdidas!   EXEMPLO: desativar compressão para Step Tests na planta.

Em geral estas análises não costumam ser feitas com dados “muito antigos”.

Verificar se o mecanismo de compressão está ativo antes da coleta de dados em testes que se pretende estudar a dinâmica de um processo qualquer, é de fundamental relevância.

Nos casos em que está ativo o mecanismo de compressão com descarte de dados, é comum na etapa de planejamento de experimento, entrar em contato com todos os responsáveis pelo Sistema de Informação, bem como com os operadores da planta para poder realizar o experimento da melhor maneira possível (segurança, produtos dentro das especificações, etc.), coletando os dados "brutos" (denominado também de raw data, em inglês ) sem a aplicação de qualquer algoritmo que altere as informações originais.

Logicamente que estes dados "brutos" terão que passar por uma etapa de tratamento antes de ser possível utilizá-los nas análises subsequentes. Frequentemente deve-se proceder com o tratamento de outliers e a remoção de ruídos de medida através de uma operação de filtragem destes dados.

Consultas à variáveis de processo

Tipicamente uma função do tipo Historical_Data = get_data(params)

  • Nome da variável
  • Data do início do período (inclusive)
  • Data do final do período (exclusive)
  • Timezone
  • Tipo de Agregação (= processamento)
  • Intervalo de Processamento
  • Tratamento das extremidades
  • ...

Todas as consultas a dados históricos de processo seguem um mesmo padrão, ou seja, é preciso informar a variável que se deseja buscar os dados, o período da consulta e, eventualmente, se é para aplicar algum processamento/cálculo adicional aos dados, como por exemplo: interpolação, média podenderada ao longo do tempo, máximos, mínimos, totalizações, etc.

No padrão OPC UA estes processamentos são designados por "agregações", ou seja, uma forma específica de agrupar os dados em informações mais relevantes, através de algum processamento matemático sobre os mesmos.

Por se tratar de Séries Temporais, é fundamental ter especial atenção com a questão dos Fuso Horários (Timezone, do inglês), por conta disso é que os softwares PIMS armazenam dados em UTC (Coordinated Universal Time), isso quer dizer que ele guarda os Timestamps contínuos, sem sofrer os problemas decorrentes de fuzo horários distintos e/ou horários de verão. Portanto, ao se efetuar uma consulta aos dados de processo, é preciso estar ciente disso a fim de informar o período de interesse corretamente.

Consultas a Dados Históricos

- Padrão OPC UA -

  • RAW
  • Interpolative
  • Average
  • Time Average
  • Total
  • Minimum/Maximum
  • Range
  • Count
  • Duration In State Zero/Non-Zero
  • ...

O padrão OPC UA define um conjunto bastante amplo de formas de aplicar agregações (=processamentos) aos dados. Recomenda-se consultar o padrão para compreender melhor os detalhes de cada tipo de consulta, onde podem ser aplicadas, etc.(OPC Foundation).

Em geral, as mais utilizadas em consultas a Dados de Processo são a Interpolative e a TimeAverage. Para variáveis do tipo on-off, as consultas sobre a duração em estado zero ou não zero são mais comuns, simplificando bastante o processo de obtenção de informações como o tempo de um equipamento ligado, por exemplo.

Vale ressaltar que os Sistemas de Informação podem estender estas consultas, oferecendo outras formas especiais de agregação. No caso do Elipse Plant Manager (EPM), foram adicionadas à lista de consultas disponíveis as seguintes: Trend, PercentTimeInStateZero e PercentTimeInStateNonZero.

Vamos fazer na prática um plugin e ver como é fácil?! ☺

A criação de um plugin em linguagem Python no ambiente Dataset Analysis do EPM Studio é bastante simples, consiste basicamente de:

  1. Criar um diretório com o nome que se deseja para o plugin (no diretório ... My Documents/Elipse Software/EPM Studio/Plugins).
  2. Colocar um arquivo de imagem no diretório do plugin para ser utilizado como ícone na ribbon.
  3. Criar um script Python carregando o módulo de Plugins do EPM, utilizando o decorator de plugins para todas as funções que deverão ser disponibilizadas ao usuário.
  4. As docstrings das funções são automaticamente utilizadas como tooltips das mesmas na ribbon.
O módulo EPM Python Plugin oferece os seguintes recursos de integração:
  • Acesso aos dados das penas selecionadas
  • Acesso às informações das variáveis das penas selecionadas (unidade de medida, descrição, etc.)
  • Janelas de: Informação, Alerta e Erro
  • Janela dinâmica de input para entrada de dados de usuários
  • “Plotar” no próprio gráfico de tendências do EPM Dataset Analysis
  • Retornar variáveis para a área de trabalho

Olha só a "cara" de uma função no plugin:


                    import Plugins as ep

                    @ep.DatasetFunctionPlugin (‘Nome no menu', 1, 'y')
                        def myFunc():
                        """ Descrição que aparecerá automaticamente no tooltip.
                        Atenção para caracteres especiais e acentos!
                        """
                        #
                        # Aqui vem o código
                        #

                        y = None

                        return y
